# Audit — musicians.jazzlore.com design fidelity & polish

**Date:** 2026-05-21
**Target:** `https://musicians.jazzlore.com` (live production, post Group D)
**Method:** Playwright MCP (Chromium), measured against the live site. Read-only — no fixes proposed.
**Output discipline:** every finding carries (1) what, (2) severity rationale, (3) layer hypothesis [FRONTEND / DATA / DESIGN], (4) measured evidence.

## Findings

### CRITICAL — breaks usability or looks broken

#### CRIT-A · Autosuggest surfaces duplicate Discogs stubs above canonical curated musicians

**What.** Typing `mile` in the home search returns 6 suggestions. The top result is **"Miles Davis + 19"** with monogram "M1". Pressing Enter navigates to `/musicians/discogs:1566346` — a separate Discogs stub entity (only 14 mosaic tiles, 1 record, h1 = "Miles Davis + 19"), not the canonical `wikidata:Q93341` Miles Davis. The canonical Miles Davis does not appear in the top 6 results for `mile`.
**Severity rationale.** A first-time visitor searching "mile" expects Miles Davis the icon — instead they land on a stub. Breaks the primary discovery flow advertised by the home headline ("Step into a musician").
**Layer hypothesis.** DATA (Aura has un-deduplicated discogs entities living alongside wikidata canonical) + FRONTEND (search index does not boost canonical-12 / does not prefer wikidata over discogs / does not collapse duplicates).
**Evidence.** Page title after Enter: `"Miles Davis + 19 — Jazzlore"`. URL: `/musicians/discogs%3A1566346`. h1 text: "Miles Davis + 19". Mosaic count: 14 (probably padding), records: 1. Screenshot: `audit-2026-05-21/dupe-miles-davis-plus-19.png`.

#### CRIT-B · Autosuggest ranking does not boost canonical curated 12

**What.** Typing `tim` returns: Tim McDonald, Tim Cansfield, Tim Kennedy, Tim Caister, Tim Mead, Timothy Hugh. **Bobby Timmons is not in the top 6.** Even at 4 chars (`timm`), Bobby Timmons is ranked #3, behind two obscure pianists "Timmy Kwebulana" and "Timmy Tappan".
**Severity rationale.** The home page explicitly markets the curated 12 ("Twelve carefully-chosen starting points"). Search should reinforce that curation. It does the opposite — pushes curated musicians below random sidemen.
**Layer hypothesis.** FRONTEND (search index or ranker has no curated-list boost) + possible DATA (curated flag may not be stored in the search index).
**Evidence.** `tim` results: `["TMTim McDonald","TCTim Cansfield","TKTim Kennedy","TCTim Caister","TMTim Mead","THTimothy Hugh"]`. `timm` results: `["TKTimmy Kwebulana","TTTimmy Tappan","BTBobby TimmonsPIANO","MTMatthew Timm","LSLudwig Stimmler","CTColin Timms"]`. Bobby Timmons is in the curated 12 (per `data-enrichment-2026-05-20.md`).

#### CRIT-C · Desktop graph is a hairball for icon-tier musicians

**What.** The desktop force-graph renders **every** collaborator with no node-count cap, no proximity filter, no LOD. For Wayne Shorter: 1,562 SVG circles, 1,257 labels, 520 collaborators, in an 800×800 px panel. Miles Davis: 1,304 circles, 1,067 labels, 434 lines, 434 collaborators (shown as 800×800 panel via `viewBox="0 0 600 500"`). Sample of first 100 circles produced **4,950 / 4,950 pairwise overlaps** (i.e., every pair within the sample fails the no-overlap check); first 100 labels produced 63 bounding-box overlaps. Inner node radii average 11.64 px; with 520 nodes in 600×500 SVG units that's ~577 sq-units per node — physically below the minimum legible spacing.
**Severity rationale.** The graph view is the signature "explore who they played with" affordance for desktop users. At icon density it is unreadable — labels stack, nodes pile, and the simulation does not converge to anything meaningful. For sparse musicians the same view is clean (Antoine: 50 circles, 16 labels — perfectly fine), so the bug is dose-dependent on collaborator count.
**Layer hypothesis.** FRONTEND (no node cap; missing LOD/clustering/filter; force-simulation tick count is fixed regardless of node count per `apps/musicians/src/features/graph/layout.ts`).
**Evidence.**

- Miles graph: 1,304 circles, 1,067 texts, 434 lines; viewBox 600×500, rendered 800×800; label font-size 9px Geist Mono.
- Wayne graph: 1,562 circles, 1,257 texts, 520 lines.
- Bobby graph: 311 circles, 288 texts, 103 lines (cluttered but legible).
- Antoine graph: 50 circles, 36 texts, 16 lines (clean).
- Label samples: `["MD","Miles Davis","trumpet","PC","Paul Chambers","double bass",...]` — 3 labels per node (initials + name + instrument), so node count ≈ texts / 3.
- Screenshots: `graph-miles-desktop-light.png`, `graph-miles-desktop-dark.png`.

### HIGH — looks broken / strong deviation from polish

#### HIGH-1 · Curated home grid is 2-column at desktop 1280px (cards are huge)

**What.** The "Twelve to begin with" grid renders **2 columns of 620 × 481 px cards** at a 1280 viewport. The spec implies a 4-column layout at md+ breakpoint (per `apps/musicians/docs/specs/musicians.md`). Cards are 4× larger than intended; the home page scrolls 6 viewport-heights on desktop and 5,336 px (~14 viewports) on mobile to walk through 12 musicians.
**Severity rationale.** The home is the front door of the app. Oversized cards push every musician off-screen and force scroll where the spec asks for a glanceable wall.
**Layer hypothesis.** FRONTEND — likely a Tailwind class missing or grid CSS rule scoped wrong; the journey row above it correctly renders 3 columns of 413 px each, so the responsive system works in general.
**Evidence.** `getComputedStyle(ul).gridTemplateColumns === "622px 622px"` at 1280×800. Card figure rect: `{w: 620, h: 481}` (aspect 1.289 ≈ 4:3.1). Mobile (390×844) correctly renders 1 column at 392 × 310 px. Screenshots: `home-desktop-light.png`, `home-desktop-dark.png`.

#### HIGH-2 · Monogram extraction bug — "Miles Davis + 19" → "M1"

**What.** The duotone-initials generator produces **"M1"** instead of "MD" (or even "M+"/"MV") for the entity "Miles Davis + 19". Implementation appears to walk space-delimited tokens and take the first character of each, with no filter for non-letter tokens — so "+19" contributes "1".
**Severity rationale.** Visible everywhere this stub is referenced; immediately reads as broken in autosuggest and on the duplicate detail page.
**Layer hypothesis.** FRONTEND (initial-extraction helper in Duo3 or the search-index renderer).
**Evidence.** First option HTML: `<span class="duo3-initials" aria-hidden="true">M1</span>` ; display text: `"Miles Davis + 19"`. By contrast, "Tim McDonald" → "TM" (correct), "Bobby Timmons" → "BT" (correct), "Nguyên Lê" → "NL" (correct).

#### HIGH-3 · "More about X →" link is an in-page anchor, not the spec's MoreAboutSheet

**What.** On Miles, Bobby, Antoine, Wayne the "More about Antoine →" / "More about Miles →" affordance is `<a href="#about">`, scrolling to an in-page anchor. The spec describes `MoreAboutSheet.tsx` — a bottom sheet with 280 ms slide, swipe ≥ 80 px dismiss, focus trap, body portal. That sheet UI is not invoked on any of the four detail pages I tested.
**Severity rationale.** Documented design surface either not built or replaced silently. The anchor scroll works, but it isn't the sheet experience the spec promises and there's no swipe-dismiss / portal behavior in play.
**Layer hypothesis.** FRONTEND (sheet component may exist but is unwired) or DESIGN (pivot from sheet to anchor scroll that wasn't reflected in the spec).
**Evidence.** All four detail pages show `<a href="#about">More about <Name> →</a>` in the Biography section. No `MoreAboutSheet` mount node in the snapshot tree.

#### HIGH-4 · Duplicate canonical-vs-stub entities are reachable as full pages

**What.** `wikidata:Q93341` (Miles Davis) and `discogs:1566346` ("Miles Davis + 19") are both fully navigable detail pages with their own h1, mosaic, records strip, etc. — sharing very similar names but presenting as different artists. No "see canonical / merge" indicator on the stub.
**Severity rationale.** Even if CRIT-A (search ranking) is fixed, deep links and journey/random pages can still land on the stub. Damages trust ("is this the same Miles Davis or a different one?").
**Layer hypothesis.** DATA primarily (Aura should mark these as alias/sameAs to canonical), with FRONTEND fallback (detect duplicate-name pairs and surface "See also: canonical Miles Davis").
**Evidence.** Both pages return 200 and render `<h1>`s. Stub page: 14 mosaic tiles, 1 record. Canonical: 14 mosaic tiles, 113 records. URLs: `/musicians/wikidata:Q93341` and `/musicians/discogs:1566346`.

#### HIGH-5 · Era strip is 100 % monogram (no portraits)

**What.** Era strip across Miles, Bobby, Wayne renders 12 era peers, **0 of 12 with photos**. The strip is exclusively duotone gradients with monogram text. Spec calls EraStrip a "sideways serendipity" surface — at zero photo coverage it reads as a wall of placeholder tiles.
**Severity rationale.** A core discovery surface presents as visually empty, despite many of the rendered peers being well-photographed (e.g., contemporaries of Miles in the late '50s include musicians whose portraits we already have in Aura — Coltrane, Mingus, etc.).
**Layer hypothesis.** Likely a **DATA pipeline gap** (era-peer query doesn't join the portrait_url field) OR **FRONTEND rendering** (era tile component intentionally suppresses the img even when portrait data is on the peer). Worth checking the BFF response shape for `/api/musicians/:id` era-peer payload.
**Evidence.** `document.querySelectorAll('.era-tile').length === 12` and `document.querySelectorAll('.era-tile img').length === 0` on all four enriched detail pages.

#### HIGH-6 · Records strip is 100 % duotone tiles — design intent or oversight?

**What.** Miles' records strip has 113 tiles, **0 with cover art**. Wayne: 104 tiles, 0 covers. Bobby: 48 tiles, 0 covers. Each `.rec-tile` is `<a class="rec-tile" href="…spotify search…"><figure class="fig"><div class="duo3 flat" …></div>title/artist/year</figure></a>` — by design renders a per-record duotone gradient and the title/label/year as text. At 100+ tiles per icon-tier musician the strip is repetitive: 113 identical-shaped duotone rectangles.
**Severity rationale.** This is a key emotional moment on a detail page ("what did they record?"). Even if cover art licensing is the blocker, the lack of visual variety at this scale is a polish hit. **Flag for design call** — strip may need a per-record texture, a contrasting motif, or a hard cap with "Show more".
**Layer hypothesis.** DESIGN (intentional duotone-only design choice; spec mentions duotone "frozen token layer") — but at icon density the design is showing strain. Could also be DATA if cover-art URLs are licensable but not populated.
**Evidence.** `recordTileSize === { w: 124, h: 244 }` on Miles desktop, `withCover === 0`. HTML sample: `<a class="rec-tile" href="https://open.spotify.com/search/..."><figure class="fig"><div class="duo3 flat" style="--duo-lo: #2a261a; --duo-hi: #9a8a4f;"></div>…</figure></a>`.

#### HIGH-7 · Font preload warnings fire on every navigation

**What.** Every page load logs three `[WARNING]` lines: `geist-400.woff2`, `geist-600.woff2`, `newsreader-500.woff2` "preloaded using link preload but not used within a few seconds from the window's load event." Six warnings on the first SPA navigation, three more per subsequent navigation (cumulative 9 by the third detail page).
**Severity rationale.** Indicates either (a) the preload `as=`/`type=`/`crossorigin=` attribute is mismatched, so the browser preloads but can't reuse the cached resource for the actual font request, or (b) the preloaded variant doesn't match the font-display rule the CSS later uses. Either way the preload is **wasted bandwidth on every nav**, and the user sees FOUT until the un-preload-able request settles.
**Layer hypothesis.** FRONTEND (build pipeline / `<link rel=preload>` head injection).
**Evidence.** `[WARNING] The resource https://musicians.jazzlore.com/fonts/geist-400.woff2 was preloaded using link preload but not used within a few seconds from the window's load event.` and 2 sibling warnings, repeating on each nav.

### MEDIUM — deviation from polish

#### MED-1 · Mobile journey row squishes to three 116-px columns

**What.** At 390 × 844, the "Start a journey" row renders as `grid-template-columns: 116.664px 116.664px 116.664px` — three CTAs at 116 px each. Each CTA contains an icon, a headline ("Random jump" / "Era walk" / "Label walk"), and a one-line description. At 116 px the description wraps 4–5 lines and the click target is cramped.
**Severity rationale.** Discoverable but tight; should consider stacking the three CTAs vertically on mobile.
**Layer hypothesis.** FRONTEND/DESIGN.
**Evidence.** Computed columns above; screenshot `home-mobile-light.png`.

#### MED-2 · Autosuggest options have no aria-label; screen reader gets concatenated text

**What.** Each `<li role="option">` renders `<span class="duo3-initials" aria-hidden="true">M1</span>` + `<div class="nm"><em>Mile</em>s Davis + 19</div>` with no `aria-label` on the option itself. Voice readers will announce "Miles Davis + 19" with no semantic separator between the visually decorative initials and the name; the highlight `<em>` further fragments the read-out ("Mile … s Davis plus 19").
**Severity rationale.** Discoverability for vision-impaired users; project-wide a11y bar is "0 axe violations, AA contrast" — this fails the spirit if not the letter.
**Layer hypothesis.** FRONTEND.
**Evidence.** `items.map(i => i.getAttribute('aria-label')) === [null, null, null, null, null, null]` on the `mile` query.

#### MED-3 · Detail page scroll height is heavy

**What.** Miles desktop scroll = 3,293 px (~4 viewports). Mobile = 3,388 px. Antoine mobile (sparse) = 3,162 px — almost as tall as Miles, due to the 14 mtiles + 16 collab rail rendering regardless of pluck count. Home mobile = 5,336 px.
**Severity rationale.** Once HIGH-1 is fixed the home will shrink, but detail pages will still feel long because mosaic + collab rail render aggressively even on sparse musicians.
**Layer hypothesis.** FRONTEND/DESIGN — possibly the rail expansion + mosaic should shrink for sparse data instead of padding to 14/16.
**Evidence.** `document.documentElement.scrollHeight` values above.

#### MED-4 · Mosaic pads to 14 tiles even when musician has very few collaborators

**What.** Antoine Hervé renders 14 mosaic tiles even though only 14 collaborators exist (no "Show all X" button — i.e., these ARE all of them). With only 1 of the 14 having a photo, the orbit looks like 13 monogram swatches around 1 portrait — closer to a placeholder grid than a curated mosaic.
**Severity rationale.** Sparse musicians look starkly empty without the empty-state copy that the hero gets ("No portrait on file — Wikimedia Commons request pending.").
**Layer hypothesis.** DESIGN (orbit cap rule of 14 ignores the photo-coverage signal) + DATA (low photo coverage for French sidemen).
**Evidence.** Antoine: `{mosaic: {total: 14, withPhoto: 1}}`, `totalCollabsAria: null` (no "Show all").

#### MED-5 · Hero fallback caption is inconsistent

**What.** Antoine's hero shows a duotone block with caption "No portrait on file — Wikimedia Commons request pending." Bobby Timmons' hero is also photo-less — but has **no caption**. So the user gets context for one missing portrait and silence for another.
**Severity rationale.** Inconsistent affordance — the empty state design is good when it shows, but it doesn't show on all photo-missing detail pages.
**Layer hypothesis.** DATA (the "request pending" copy is probably keyed off a `portrait_pending` flag that's set for Antoine but not Bobby) or FRONTEND (caption logic).
**Evidence.** Antoine snapshot: `figure "No portrait on file — Wikimedia Commons request pending."`. Bobby snapshot: no equivalent caption on the hero.

#### MED-6 · Autosuggest doesn't show portraits for musicians that have them

**What.** All 6 autosuggest options render the duotone-initials swatch, never the actual portrait — even for canonical curated musicians where the photo is available in Aura.
**Severity rationale.** A photo-rich suggestion would visually disambiguate "Miles Davis" from "Miles Bould" instantly. Without it, the user falls back to reading 4–6 monogram tiles.
**Layer hypothesis.** FRONTEND (Autosuggest component renders only initials; spec said 80 ms debounce, ≤ 6 results — design did not call for portrait, possibly intentional).
**Evidence.** `home-desktop-light.png` autosuggest open state shows 6 duotone tiles, no `<img>` inside any option.

#### MED-7 · Mosaic tile aspect ratios produce strip-like (148 × 44) tiles

**What.** On Miles desktop, the 14 mosaic tiles span sizes from 73 × ?? to 224 × 138. The skinny variant is 148 × 44 px (3.36:1 aspect) — barely tall enough for the monogram + instrument label.
**Severity rationale.** Cosmetic but reads as "broken layout" at a glance. The size encoding (records → tile size) is a strong idea but the skinny s3 size hits a legibility floor.
**Layer hypothesis.** DESIGN (size formula in `MosaicV4.tsx` produces a sub-min-height tile).
**Evidence.** `mosaicSizes` sample at Miles desktop light: `[{w:224,h:138},{w:148,h:91},{w:148,h:44},{w:148,h:44},{w:148,h:44}]`.

### LOW — polish, nice-to-have

#### LOW-1 · `body` background is `rgba(0, 0, 0, 0)` (transparent)

**What.** `getComputedStyle(body).backgroundColor === "rgba(0, 0, 0, 0)"` on every page. The theme bg is painted via `<html data-theme="…">` or a wrapper. Works visually today but is brittle if a portal/overlay renders into `<body>` directly (e.g., the future MoreAboutSheet from HIGH-3).
**Layer hypothesis.** FRONTEND/DESIGN.
**Evidence.** Probed on home + 4 detail pages, all themes.

#### LOW-2 · "Listen to X" section only shows Spotify + Apple Music — no preview/embed

**What.** The "Listen to Antoine Hervé" / "Listen to Miles Davis" section is a pair of outbound CTAs (`open.spotify.com/search/…`, `music.apple.com/search?term=…`). No preview, no inline audio. Consistent across musicians.
**Severity rationale.** Below polish bar — this is the only place the app _says_ "Listen" and never lets you listen.
**Layer hypothesis.** DESIGN — likely intentional v1 scope cut.
**Evidence.** Antoine snapshot shows the two outbound `<a>`s; Miles same.

#### LOW-3 · No skip-to-content link

**What.** Banner → main pattern but no `<a class="skip-to-content">` for keyboard users.
**Layer hypothesis.** FRONTEND/A11Y.
**Evidence.** Snapshot tree on home: `banner` then `main` with no skip-link in between.

#### LOW-4 · Inconsistent "More about" → `#about` target (no anchor verified)

**What.** All four detail pages use `href="#about"`. I did not verify whether an element with `id="about"` actually exists; if not, the anchor click jumps to top-of-page (default browser behavior).
**Layer hypothesis.** FRONTEND.
**Evidence.** `moreAboutHref === "#about"` on Miles, Bobby, Antoine, Wayne. Not probed for target element presence.

#### LOW-5 · Cover-art `<img>` count anomaly on Miles desktop

**What.** Miles' main element has 31 `<img>` tags (1 hero + 14 mosaic + 16 collab-rail headliners = 31 — consistent), but the rec-tile + era-tile sections render zero images, so the 31 are entirely portrait-related, none album-art-related. Confirms HIGH-5 / HIGH-6 from another angle.
**Layer hypothesis.** N/A — already covered.
**Evidence.** `document.querySelectorAll('main img').length === 31`, alts cluster as `["Miles Davis", "Paul Chambers", … duplicated …]`.

## What this audit deliberately did NOT do

- No root-cause investigation beyond a layer hypothesis per finding.
- No code edits, no commits.
- No Lighthouse/axe-core full run (separate concern; the project has `pnpm lighthouse:audit` and a committed `tests/e2e/a11y.spec.ts`).
- Did not test the journey landing pages (`/journey/random`, `/journey/era`, `/journey/label`) — out of original scope ask.
- Did not exercise mobile autosuggest (desktop only).
- Did not test focus traps or swipe gestures on a hypothetical `MoreAboutSheet` because the sheet did not appear to be wired up (see HIGH-3).
- Did not measure under throttled CPU/network — these are static-content observations on a fast connection.
